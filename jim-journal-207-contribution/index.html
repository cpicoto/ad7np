WW7ATS DATV transmit/receive software toolchain

Hi Jim and BATVC readers,

For WW7ATS we run a DATV repeater workflow that couples RF delivery to an operator-controlled internet distribution pipeline.

The stack has three roles:

1) RF ingest & control (HiDesReceiver)
   A DVB-T DATV receiver/control program that sets the RF/DATV parameters and maintains a stable feed for downstream streaming.
   In practice, it gives operators an immediate “signal vs no-signal” view and dependable control behavior during short sessions and RF dropouts.

2) Operator transmit/receive app (WW7ATS)
   A native app (iOS/iPadOS/macOS) that lets an operator:
   - watch the incoming feed (native low-latency HLS playback)
   - start/stop an RTMP transmit session with one large PTT control
   - see clear connection-state UX (“connecting”, “ON AIR”, “failed”) and a simple retry action
   Stream keys are stored in the OS Keychain (no plain-text persistence), and the app uses audio/session handling to avoid feedback while transmitting.

3) Cloud headend (Azure VM: MediaMTX + nginx)
   MediaMTX accepts RTMP/RTMPS ingest, remuxes to low-latency HLS for viewers, and provides runtime hooks + an API for monitoring.
   The headend sends connection events on connect/disconnect to the legacy WW7ATS repeater dashboard so net control can verify when a transmit session is actually live.

How we operate (field workflow):

- Before transmitting, operators verify RF presence using the spectrum/signal monitor view (Figure 1).
- The operator prepares the OBS scene/camera and taps PTT.
- The WW7ATS status chip confirms the session state; if anything fails, the operator retries without re-entering credentials.
- Net control watches the connection events and monitoring signals to confirm “signal vs no-signal” and that the correct path is publishing.

This design was built for field reality: short operator sessions, occasional RF transitions, and minimal friction for licensed amateurs.

73 de Carlos (AD7NP)
